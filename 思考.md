

  1. 流程设计问题

  - PRD -> plan -> code edit -> test -> fix -> result 
  这个流程缺少了"验证计划"的人工确认环节
  - 没有回滚机制 - 如果修改出问题，如何回退？

  2. 风险控制

  - "max 2 repair loops" -
  这个数字是否合理？没有解释为什么是2
  - "stop on unsafe or ambiguous situations" -
  但没有定义什么是"unsafe"或"ambiguous"

  3. 测试策略

  - "one fixed test command" - 这可能导致：
    - 全量测试太慢
    - 无法针对性测试修改的部分
    - 可能触发无关的 flaky tests

  4. 上下文收集

  - "Do not dump the whole repository into the prompt" -
   但如何确定"relevant files"？
  - 如果 file_hints 不准确或不完整怎么办？

  5. 编辑格式

  - SEARCH/REPLACE blocks - 但如果 SEARCH
  文本在文件中有多个匹配怎么办？
  - 没有处理冲突的情况

  6. 状态管理

  - 状态模型很简单，但缺少：
    - 错误历史
    - 中间检查点
    - 回滚点

  7. 安全考虑

  - 没有 API key 管理
  - 没有 rate limiting 考虑
  - 没有审计日志

  Technical Design 分析

  1. 脚本接口设计

  - 所有脚本用 JSON 输入/输出很好，但：
    - 没有 schema validation
    - 没有版本控制
    - 错误处理不够详细

  2. apply_patch.sh

  - "bounded fuzzy matching" -
  模糊匹配存在风险，可能导致错误应用补丁
  - 需要更严格的匹配策略和验证机制
  - 缺乏对大型补丁的处理方案

  3. run_tests.sh

  - 分类测试方法存在不确定性
  - "repairable" 判断逻辑不够精确
  - 输出截断可能丢失关键调试信息
  - 测试分类依赖于脆弱的关键词匹配

  4. 上下文管理

  - 文件行数限制可能意外排除关键文件
  - 跳过文件的策略不够灵活
  - 大文件处理方案单一且不可靠

  5. 工作流设计

  - n8n 节点缺乏持久化机制
  - 错误恢复能力薄弱
  - 中断后无法自动恢复

  6. 分支策略

  - 分支名称生成方法过于僵化
  - 并发执行存在潜在风险
  - 缺乏并行任务支持

  7. 日志管理

  - 日志存储位置未明确
  - 缺乏日志轮转策略
  - 未考虑长期存储需求

  8. 配置文件

  - 配置缺乏验证机制
  - 敏感信息处理不当
  - 配置更新需要重启

  9. LLM 调用

  - 未定义调用失败处理方案
  - 缺少超时和重试策略
  - 费用监控存在盲点

  10. Prompt 设计

  - 模板变量替换规则不清晰
  - 未考虑 token 限制
  - 输出格式缺乏严格验证

  关键问题需要系统性解决，确保工作流健壮性和安全性。

⏺ 我来深度分析这两份文档的漏洞。                        
 
  🔴 核心架构漏洞                                       
                                                        
  1. 缺少"人工确认检查点"                               
                                                        
  PRD 第 72 行明确说 "keeping final review in human   
  hands"，但整个流程只有在最后才有人工审查：
  PRD -> plan -> code edit -> test -> fix -> result
                            ↑
                      这里应该有人工确认

  风险：LLM 可能产生看似合理但实际错误的 plan，直接进入
  code edit 阶段浪费资源。

  修复：在 plan 阶段后增加可选的 --dry-run 或 human
  approval step。

  ---
  2. 没有回滚机制

  Technical Design 第 533-544 行描述了 branch 策略，但：
  - 没有 git stash 或 git reset 的自动化
  - 如果 apply_patch.sh 部分成功、部分失败，如何恢复？
  - repair loop 失败后，repo 处于什么状态？

  修复：每个 apply_patch.sh 前创建 git tag 作为
  checkpoint。

  ---
  3. SEARCH/REPLACE 格式的致命缺陷

  PRD 第 156 行说 "SEARCH/REPLACE blocks"，但：

  问题 A：如果 SEARCH 文本在文件中出现多次怎么办？
  <<<<<<< SEARCH
  function hello() {
  =======
  function goodbye() {
  >>>>>>> REPLACE
  如果文件里有 3 个 function hello() {，替换哪个？

  问题 B：模糊匹配的边界条件？
  Technical Design 第 277 行说 "bounded fuzzy
  matching"，但：
  - 多少字符差异算"匹配"？
  - 空白字符差异如何处理？
  - 这些参数是硬编码还是可配置？

  修复：强制要求 SEARCH 文本包含足够上下文（如前后 3
  行），或使用行号锚定。

  ---
  🟡 测试策略漏洞

  4. "Fixed Test Command" 的隐患

  PRD 第 170-180 行：
  Run one fixed command only, for example: npm test

  问题：
  - 全量测试可能触发无关的 flaky tests
  - 没有考虑 test filtering（如 pnpm test -- --grep
  "users"）
  - 如果测试超时怎么办？没有 timeout 配置

  修复：增加 test_timeout 配置项，支持
  test_filter_pattern。

  ---
  5. Failure Classification 的脆弱性

  Technical Design 第 563-580 行用关键词判断 failure
  type：
  - contains `eslint` -> `lint`
  - contains `TS2322` -> `type`

  问题：
  - 如果错误信息包含多个关键词怎么办？
  - 不同框架的错误格式不同（Jest vs Vitest vs Mocha）
  - "non-repairable" 的判断逻辑过于简单

  修复：使用正则表达式或结构化解析（如 tsc --pretty
  false），而非简单关键词。

  ---
  🟠 上下文收集漏洞

  6. max_file_lines_for_context 的粗暴截断

  Technical Design 第 601-606 行：
  if a selected file exceeds max_file_lines_for_context,
   skip it

  问题：
  - 一个 1001 行的文件被完全跳过，但可能关键代码在最后
  10 行
  - 没有提供"只读取部分内容"的选项

  修复：允许 file_hints 指定行号范围：
  {
    "file_hints": [
      {"path": "src/large.ts", "lines": "100-150"}
    ]
  }

  ---
  7. 隐式依赖未被检测

  如果 file_hints 提供了 src/routes/users.ts，但该文件
  import 了 src/utils/validation.ts：
  - LLM 不知道 validation.ts 的内容
  - 可能产生与现有验证逻辑冲突的代码

  修复：自动收集 import 依赖链（深度可配置，如 1-2
  层）。

  ---
  🔵 并发与状态漏洞

  8. 并发执行未考虑

  Technical Design 第 130 行的 branch naming：
  "branch_name": "agent/task-20260312-001"

  问题：如果同一时刻有多个 task：
  - 同一 repo 上的并发 branch 操作冲突
  - n8n 的 state 是 per-execution 还是 global？
  - 没有锁机制

  修复：在 check_repo.sh 中使用 git lock 或文件锁。

  ---
  9. 状态持久化缺失

  Technical Design 第 100-141 行描述了 state shape，但：
  - 状态存在哪里？（内存？数据库？文件？）
  - 如果 n8n crash，状态丢失吗？
  - 如何查询历史执行记录？

  修复：明确指定状态存储后端（如 SQLite 或 Supabase）。

  ---
  🟣 LLM 调用漏洞

  10. 没有 token 限制和成本控制

  Prompt 模板（第 391-513 行）没有：
  - max_tokens 配置
  - temperature 设置
  - 模型版本选择（GPT-4 vs GPT-3.5）
  - 费用预算上限

  修复：在 config 中增加：
  {
    "llm": {
      "model": "claude-sonnet-4-6",
      "max_tokens": 4000,
      "temperature": 0.2,
      "max_cost_usd": 1.00
    }
  }

  ---
  11. 输出格式校验缺失

  Prompt 要求返回 JSON，但没有：
  - JSON schema validation
  - 处理 malformed JSON 的 fallback
  - 验证必需字段（如 target_files, change_strategy）

  修复：在 LLM response 后增加 validation step，失败则
  retry 或 stop。

  ---
  ⚫ 安全漏洞

  12. 命令注入风险

  Technical Design 第 186-206 行的脚本接口：
  {
    "test_command": "pnpm test"
  }

  问题：如果 test_command 被篡改为 rm -rf / && pnpm
  test？

  修复：白名单验证，只允许预定义的命令模式。

  ---
  13. 敏感信息泄露

  - LLM prompt 可能包含代码中的 secrets
  - 没有自动扫描 .env 或 secret patterns

  修复：在 collect_context.sh 中增加 .env 和 secret*
  的过滤逻辑。
